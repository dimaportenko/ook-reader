# Ook Reader

A cross-platform (iOS / iPadOS, with macOS planned) EPUB reader written in Swift,
developed in **NeoVim** instead of Xcode. macOS comes later via Mac Catalyst (see
[`RESEARCH.md`](RESEARCH.md) for the full plan and rationale).

## Requirements

| Tool | Why | Install |
|---|---|---|
| Xcode 16+ (we use 26.5) | SDKs, simulators, `lldb-dap`, signing | App Store |
| [Tuist](https://tuist.dev) | Generates the Xcode project from `Project.swift` | `brew install tuist` |
| [xcode-build-server](https://github.com/SolaWing/xcode-build-server) | Bridges sourcekit-lsp to the Xcode project | `brew install xcode-build-server` |
| NeoVim + the Swift config | Editing, LSP, build/run/debug | see [Editor setup](#editor-setup-neovim) |

## Project layout

```
ook-reader/
├── Project.swift              # Tuist manifest — the SOURCE OF TRUTH for the project
├── Sources/                   # app code (globbed into the OokReader target)
│   ├── OokReaderApp.swift     #   @main SwiftUI entry point
│   └── ContentView.swift      #   root view
├── Tests/                     # unit tests (OokReaderTests target)
├── RESEARCH.md                # toolchain research + phased build plan
├── README.md                  # this file
│
│  # generated / machine-local — NOT committed (see .gitignore):
├── OokReader.xcworkspace      # generated by `tuist generate`
├── OokReader.xcodeproj        # generated by `tuist generate`
└── buildServer.json           # generated by `xcode-build-server config`
```

The `.xcodeproj` / `.xcworkspace` are **build artifacts** generated from `Project.swift`.
You never edit them by hand — you edit the manifest and regenerate.

## First-time setup

From the repo root:

```sh
# 1. Generate the Xcode project from the manifest
tuist generate --no-open

# 2. Wire sourcekit-lsp to the generated project (creates buildServer.json)
xcode-build-server config -workspace OokReader.xcworkspace -scheme OokReader
```

That's it — both outputs are gitignored, so every clone runs these two commands once.

## Daily workflow

### The golden rule

> **Edit `Project.swift` → run `tuist generate`.**
> Editing files *inside* `Sources/` does **not** need regeneration (they're globbed).
> Only structural changes — new target, new dependency, changed build settings —
> require `tuist generate` (and usually re-running `xcode-build-server config`).

### From the command line

```sh
tuist generate --no-open                         # regenerate after manifest changes

xcodebuild -workspace OokReader.xcworkspace \     # build for a simulator
  -scheme OokReader \
  -destination 'generic/platform=iOS Simulator' build

xcodebuild -workspace OokReader.xcworkspace \     # run the test target
  -scheme OokReader \
  -destination 'platform=iOS Simulator,name=iPhone 16 Pro' test
```

### From NeoVim (xcodebuild.nvim)

Run **`:XcodebuildSetup`** once per machine to pick the workspace / scheme / config / device.
After that:

| Keymap | Action |
|---|---|
| `<leader>X`  | Open the action picker (all commands) |
| `<leader>xb` | Build |
| `<leader>xr` | Build & run on the selected simulator |
| `<leader>xt` | Run tests |
| `<leader>xc` | Change the selected device |
| `<leader>xs` | Change the scheme |
| `<leader>xl` | Toggle the build/run logs |
| `<leader>dd` | Build & debug (breakpoints via Xcode's `lldb-dap`) |
| `<leader>dn` | Debug without rebuilding |
| `<leader>dT` | Debug tests |
| `<leader>dx` | Terminate the debug session |
| `<leader>db`, `<leader>dc`, `<leader>di`, … | breakpoint / continue / step (LazyVim dap) |

LSP: sourcekit-lsp attaches automatically to `.swift` files. Use `:checkhealth lsp`
to inspect attached clients (the old `:LspInfo` command was removed).

## Editor setup (NeoVim)

This repo assumes a NeoVim config with the Swift toolchain wired up
(LazyVim + `sourcekit` LSP + `xcodebuild.nvim` + `nvim-dap`). The full toolchain
breakdown — and what still requires Xcode (signing, Instruments, asset catalogs) —
is documented in [`RESEARCH.md`](RESEARCH.md), Phase 0.

## Notes & gotchas

- **`No such module 'ProjectDescription'` in `Project.swift`** — harmless. That module
  ships inside Tuist and is only available when Tuist compiles the manifest; sourcekit-lsp
  can't see it. It does not affect `tuist generate`.
- **macOS target** — intentionally absent for now. The Readium rendering navigator is
  UIKit-only and its package targets iOS; macOS will arrive via Mac Catalyst in a later phase.
- **Regenerating from scratch** — safe to delete `OokReader.xcodeproj`, `OokReader.xcworkspace`,
  and `buildServer.json` at any time; the two setup commands recreate them.
